🦞 Developer how-to

How to get started with OpenClaw

This page is for developers and technical operators. The goal is not to build a huge agent zoo. The goal is to get one useful setup working in a way that is visible, cheap enough, and safe enough to iterate on.

Main exploration
Recommended first path: start VPS-first, connect one or two models, set up Telegram, prove one useful workflow, then layer in memory, routing, and additional integrations.
1. VPS-first hosting

Choose the hosting environment deliberately

There are three common ways to run OpenClaw: a personal machine, a Mac mini, or a VPS. For most developer setups, a VPS is the best first choice because it runs 24/7 and keeps the bot separated from your personal files and environment.

VPS

Best default for most people. Always on, cleaner for remote access, easier to isolate from your daily machine.

Recommended starting point

Mac mini

Good for a more controlled internal host, especially if you want tighter physical/runtime control.

Personal machine

Fine for proving the idea, weak for trust, uptime, security, and team use.

Cost tip: KVM1 is enough for API-based workflows. Use KVM4 or similar only if you plan to run local models like Ollama.
2. Connecting models

Start with Anthropic and OpenAI, then add routing

Anthropic

  • add an Anthropic API key
  • the setup guidance recommends funding enough to reach Tier 2
  • that improves rate limits substantially and makes setup less frustrating

OpenAI

  • add an OpenAI API key
  • set a monthly spend cap in the provider dashboard
  • do this before experimentation gets messy
Safety net: set a monthly limit early (for example $100) so mistakes do not become surprise bills.
3. Telegram setup

Telegram is still one of the best first interfaces

Use @BotFather. Create a bot, give it a name, and choose a unique username ending in bot.
Add the API token to OpenClaw. Then message the bot yourself.
Pair your account. The pairing code adds your Telegram ID to an allow list so only authorised users can talk to the assistant.
4. Google Workspace

This is the tedious bit

Expect this part to take around 10–15 minutes. It is not hard exactly; it is just annoying.

What you need to do

  • create a Google Cloud project
  • enable Gmail, Calendar, Drive, and any other needed APIs
  • configure the OAuth consent screen
  • download the client secret JSON

How the pairing works

  • send the client secret JSON to the bot
  • follow the auth link
  • authorise access
  • test both read and write actions
Good test: add a dummy calendar event and then ask the bot to read it back. Verify both directions.
5. Model routing

Cheap-first routing, clear tiers, and explicit escalation

Cost control should be treated as an operating discipline, not just a model-choice preference. The goal is to reserve stronger models for genuinely difficult work while cheaper models handle routine tasks.

Illustrative pricing examples: GPT-5 mini is roughly $0.25 / $2.00 per 1M input/output tokens, Gemini 2.5 Flash is roughly $0.30 / $2.50, Gemini 2.5 Flash-Lite is roughly $0.10 / $0.40, while Claude Sonnet 4.6 is roughly $3.00 / $15.00. Premium defaults can become dramatically more expensive than cheap-first routing.

Tier 1 — cheapest / fastest

  • classification
  • tagging
  • daily digests
  • reminders
  • simple summaries
  • lightweight routing
  • inbox triage

Examples: Gemini 2.5 Flash-Lite · GPT-5 mini · Claude Haiku 4.5

Tier 2 — low-cost general work

  • project shaping
  • rough drafting
  • meeting-note synthesis
  • task breakdown
  • lightweight web / docs work
  • agent-to-agent summaries

Examples: Gemini 2.5 Flash · GPT-5 mini · Claude Haiku 4.5 / low-cost Sonnet

Tier 3 — strong default production model

  • most coding work
  • non-trivial planning
  • implementation steps
  • code review
  • debugging
  • quality-sensitive drafting

Examples: GPT-5.4 · Claude Sonnet 4.6 · Gemini 2.5 Pro

Tier 4 — premium / sparse use

  • hardest debugging
  • architecture decisions
  • difficult synthesis
  • large codebase reasoning
  • critical final passes
  • stubborn edge cases

Examples: Claude Opus 4.6 · GPT-5.4 high-reasoning · Gemini 2.5 Pro

Operational pattern: use cheap-first routing, per-agent budgets, spend tracking, context-window management, narrow contexts, structured handoffs, and resets after milestones rather than letting sessions turn into whales.
Free / cheap fallback: keep a fallback model available so the bot does not fail silently if credits run out.
6. Automation

Cron jobs vs heartbeat

Cron jobs

Use for exact timings.

  • morning email/calendar summary
  • nightly backups
  • specific scheduled syncs

Heartbeat

Use for lightweight urgency checks.

  • watch for important changes
  • avoid stuffing routine tasks into it
  • otherwise it will burn tokens constantly
7. Workspace files

The markdown files are the brain

AGENTS.md

Stable workflow rules and operational constraints.

SOUL.md

Personality, tone, and how the assistant behaves.

USER.md

Context about the person being helped.

MEMORY.md + daily notes

Long-term facts plus short-term logs of what happened.

8. Memory optimisation

Memory only works well if you maintain it deliberately

  • improve the memory-flush prompt so it captures durable facts, not noise
  • enable session indexing so past work can be found again
  • periodically curate the markdown files by hand
  • externalise important relationships rather than hoping context will carry them
Rule of thumb: memory quality matters more than memory quantity.
9. Cost optimisation

OpenClaw can be made dramatically cheaper

  • reset sessions when switching topics
  • use compaction / summarisation rather than dragging huge histories forever
  • switch heartbeats to cheaper or local models
  • reserve premium models for genuinely difficult work
  • use local Ollama models for low-value background checks if you have the setup
Reality check: heartbeats alone can become expensive if they run on premium models all month.
10. Advanced usage

Sub-agents and voice are where things get fun

Sub-agents

Useful when you want parallel research or bounded delegated work. Think: three small employees, not one bloated chat.

Voice

Whisper for transcription plus EdgeTTS or similar for output can make the bot much easier to use on the move.

11. Safety and recovery

Snapshots, stop commands, and not being reckless

  • take a snapshot before major config changes
  • keep a stop command or VPS-level kill path ready
  • treat external writes and credentials as high-risk
  • use deterministic sanitisation and restricted write permissions where possible
12. Useful first workflows

Good first things to build

Customer support

Triage, draft responses, escalate edge cases, and surface recurring themes.

Planning / workflow support

Shape requests, create tasks, route work, and reduce coordination drag.

Knowledge base

Drop articles, videos, and notes into a searchable system that can return summaries.

Technical delivery

Use only once repo access, review patterns, and failure handling are clean enough.

13. First-week checklist

The sane first week

Day 1: choose VPS, connect models, set spending limits.
Day 2: pair Telegram, verify basic messaging, test one simple workflow.
Day 3: connect Google Workspace if needed.
Day 4: set up routing and cheaper defaults.
Day 5: test one repeatable workflow end to end and decide whether it is actually useful.
Success looks like: one workflow that is genuinely useful, cheap enough to keep, and simple enough to explain.
