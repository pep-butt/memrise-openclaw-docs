OpenClaw

How the system is structured today

OpenClaw works best as a coordination and execution layer for real workflows. Instead of putting everything into one giant conversation, it separates work across dedicated agents, sessions, identities, and interfaces so requests can be routed, executed, reviewed, and monitored clearly.

What it is

A multi-agent operating layer that connects people, agents, tools, channels, and approvals.

What it changes

It reduces context sprawl, clarifies responsibilities, and makes handoffs and review points more explicit.

Why it matters

The win is not one clever assistant. The win is a system that makes real work more legible and governable.

This page is meant to explain the current operating model clearly: who the agents are, how work flows through the system, where interfaces sit, how governance works, and what principles the whole setup should be built around.

Current operating system

Why it is structured this way

This system is structured around specialised agents, separate identities, lightweight interfaces, and explicit control layers because the goal is not just to make AI available. The goal is to make it usable, governable, and effective in real workflows.

A single general-purpose agent quickly becomes overloaded. It has to hold too much context, switch between too many responsibilities, and operate across too many systems at once. That increases cost, reduces clarity, and makes failures harder to understand. By splitting work across specialised agents, each agent can operate with a narrower role, a smaller context window, and a clearer operating boundary.

Separate identities and scoped permissions are used so that agents can act independently without collapsing into one shared, opaque system. This improves auditability, makes permissions easier to reason about, and supports safer autonomy. A coordination agent and an execution agent should not have identical access or responsibilities.

The interfaces are also chosen deliberately. Lightweight interfaces such as Telegram make it easy to interact with agents quickly and on the go, while Slack provides a more shared operational layer for team visibility, approvals, and coordination. The aim is to let agents fit into existing workflows rather than forcing people into a new tool or habit.

The control layer is included because visibility is essential. Once work is distributed across multiple agents, operators need to be able to see what is running, what has completed, what is blocked, what has failed, and where intervention is needed. Without that, the system becomes a black box. With it, autonomy becomes manageable.

This structure also supports better cost control. Smaller sessions, clearer handoffs, and model routing make it possible to reserve stronger models for genuinely difficult work while cheaper models handle routine tasks. That keeps the system more sustainable and prevents complexity from turning into unnecessary spend.

Overall, the structure is designed to balance autonomy with control. Agents are able to move work forward independently, but within a model that keeps responsibilities clear, permissions bounded, costs more manageable, and humans in control where it matters.

Short version: specialised agents keep roles and context clear; separate identities improve auditability and permissions design; lightweight interfaces reduce friction; control layers make autonomous work visible and manageable; approvals preserve human judgment where risk matters; and model routing helps control cost.
Who are my agents?

From personal setup to a Memrise-ready agent model

The first part of this story is the personal setup: a coordination agent and a coding agent, each with a distinct role, identity, and operating surface. That gives a concrete proof point for how the model works in practice. The next step is not to copy Peppermint Butler and BMO literally into Memrise, but to translate the underlying pattern into agent roles that make sense for the organisation.

BMO — coding agent

Why individual accounts?

Dedicated identities make agent-based workflows easier to manage, safer to operate, and clearer to supervise.

Clear separation of roles

Each agent has a distinct function and operating surface, rather than one shared identity doing everything.

Better auditability

Actions can be traced back to the specific agent account that performed them.

Safer permissions design

Access can be scoped by workflow and role, instead of giving one general-purpose identity broad access to everything.

Supports controlled autonomy

Agents can operate independently within defined boundaries, while still escalating sensitive or higher-risk actions.

Cleaner handoffs between agents

A planning agent and an execution agent can work together without collapsing into one overloaded session or identity.

Less operational ambiguity

It is easier to understand who did what, where work lives, and which agent is responsible for the next step.

More resilient workflow design

If one agent drifts, fails, or needs to be reset, it can be managed without disrupting the whole system.

Bottom line: individual accounts turn agents from a shared black box into distinct, traceable operators.

Development use

Development workflow: agent chaining and repository-linked execution

Agent chaining means one agent does not have to do everything itself. Work can be passed between specialised agents, with each one handling a distinct part of the workflow.

In this model

  • Peppermint Butler handles scoping, coordination, and task shaping
  • BMO handles technical execution
  • other agents can be added later for review, security, support, or reporting

Why this matters

  1. It keeps context smaller. Separate agents help avoid one long, bloated conversation trying to hold planning, implementation, review, and reporting all at once.
  2. It creates clearer responsibilities. A coordinator agent can own scoping and routing while a builder agent owns execution.
  3. It makes governance easier. Clear approval rules, role separation, and bounded autonomy are easier to reason about.
Peppermint Butler handles project scoping and orchestration. It translates a request into a repository-linked GitHub project, creates the implementation task list, and structures the execution flow.
BMO handles technical delivery. It works through the queued tasks, building and refining outputs until the work is complete, while escalating blockers, ambiguities, or higher-risk actions for review.
Result. A repeatable path from idea → scoped project → tracked execution → completed work.
Interfaces

Interfaces: Telegram, Slack, and the practical command surface

Telegram

Telegram works well as a lightweight, mobile-first interface into a much more powerful system.

  • easy to spin up multiple bots for different agents or workflows
  • keeps sessions and context more contained
  • fast session reset and recovery when an agent gets stuck or drifts
  • useful on the go for quick requests, updates, and approvals
  • still powerful despite the simple surface
  • low-friction desktop companion for snippets and quick asks

Slack

Slack is where a lot of operational communication already happens, so it reduces friction by meeting people in an existing workflow rather than forcing a separate tool.

  • central place to manage multiple agents
  • useful for both direct requests and workflow updates
  • supports team visibility
  • good for approvals and human-in-the-loop workflows
  • works well with GitHub, task boards, support systems, and internal workflows
  • helpful for ongoing agent management and supervision
AI management

Control layer / mission control

One of the biggest challenges in agent-based systems is visibility. It is not enough for agents to be capable of doing useful work. Operators need to be able to see what is happening, what has already happened, and where intervention may be needed.

Mission control is not just monitoring. It is a way to supervise autonomous work without manually inspecting every conversation or workflow step. It reduces the black-box feeling that makes people distrust agentic systems.

Key advantages

  • visibility into what each agent is doing
  • active sessions and running workflows
  • task status and workflow progress
  • approvals and exceptions in one place
  • failures, retries, and stuck states are easier to spot
  • clearer auditability of actions across the system
  • easier intervention through reset, reroute, or stop controls
  • cost and usage can be monitored alongside activity
Security

Security, governance, and bounded autonomy

This model is only useful if the controls are part of the operating design rather than something bolted on later.

Key principles

  • not all actions should execute automatically
  • explicit approvals at key risk points
  • clear draft versus execute boundaries
  • escalation when requirements are unclear
  • humans remain accountable for high-impact decisions
  • autonomy should expand only where trust has been earned

Structural controls

  • dedicated identities per agent
  • least-privilege access by role
  • scoped credentials and tool permissions
  • clearer separation between orchestration and execution
  • isolated sessions and workspaces where possible
  • stronger traceability of actions taken
  • approval gates for sensitive external or high-impact actions
  • auditability
Important: this should not be sold as “autonomous agents everywhere.” It is strongest when autonomy is narrow, supervised, and earned through reliability.
Cost saving

Cost control and model routing

Cost control should be treated as an operating discipline, not just a model-choice preference. The recurring pattern is simple: stop sending every task to a premium model, add a cheap routing layer in front, keep sessions narrow, and track spend centrally through a gateway such as LiteLLM.

Tier 1 — cheapest / fastest

Use for:

  • classification
  • tagging
  • daily digests
  • reminders
  • simple summaries
  • lightweight routing
  • inbox triage

Example models: Gemini 2.5 Flash-Lite · GPT-5 mini · Claude Haiku 4.5

Tier 2 — low-cost general work

Use for:

  • project shaping
  • rough drafting
  • meeting-note synthesis
  • task breakdown
  • lightweight web or docs work
  • agent-to-agent routing summaries

Example models: Gemini 2.5 Flash · GPT-5 mini · Claude Haiku 4.5 / low-cost Sonnet path

Tier 3 — strong default production model

Use for:

  • most coding work
  • non-trivial planning
  • implementation steps
  • code review
  • debugging
  • quality-sensitive drafting
  • higher-stakes internal reasoning

Example models: GPT-5.4 · Claude Sonnet 4.6 · Gemini 2.5 Pro

Tier 4 — premium / sparse use

Use for:

  • hardest debugging
  • architecture decisions
  • difficult synthesis
  • large codebase reasoning
  • final pass on critical outputs
  • gnarly edge cases where cheaper models fail

Example models: Claude Opus 4.6 · GPT-5.4 high-reasoning · Gemini 2.5 Pro

Peppermint Butler

Default: Tier 1 or Tier 2

For:

  • daily briefings
  • meeting-note summaries
  • inbox triage
  • AI news scans
  • task routing
  • project setup drafts

Escalate to Tier 3 when: the request is ambiguous, the email is sensitive, planning gets complex, or the output needs stronger judgment.

BMO

Default: Tier 3

For:

  • coding tasks
  • refactors
  • implementation
  • debugging
  • PR support
  • technical documentation

Escalate to Tier 4 when: the bug is stubborn, the repo is large and messy, architecture is unclear, repeated attempts fail, or quality risk is high.

Clean slide copy: Tier 1 is cheapest / fastest for routing, summaries, classification, and routine ops. Tier 2 handles low-cost general work for drafting, shaping, and lightweight synthesis. Tier 3 is the strong default for coding, planning, debugging, and quality-sensitive work. Tier 4 is reserved for hard reasoning, architecture, and stubborn edge cases.

This maps cleanly onto current model positioning: GPT-5.4 / Sonnet 4.6 / Gemini Pro as strong production defaults; GPT-5 mini, Flash, Flash-Lite, and Haiku as lower-cost routing and synthesis options; Opus and higher-reasoning paths reserved for the hardest work.

Who needs this?

Target users and operating scope

This model is unlikely to be the right default for everybody in the company. It is strongest when used in targeted workflows where there is a clear owner, a repeatable process, and a meaningful coordination burden that can be reduced.

Best suited to

  • workflow owners and operators
  • product and project leads
  • support, lifecycle, or coordination-heavy roles
  • teams with repeatable, high-friction processes
  • users willing to supervise and review outputs

Less suitable as a default

  • everyone in the company by default
  • workflows with unclear ownership
  • high-risk actions without human review
  • contexts where deterministic outputs are essential
  • uncontrolled experimentation at scale

The goal is targeted leverage, not universal rollout.

OpenClaw memory

Memory should be externalized, structured, and intentionally reused

One of the practical strengths of this model is that it does not rely entirely on the model’s context window as memory. Ongoing project state can live in GitHub Projects, documentation, structured notes, and durable files, then be retrieved as needed.

Why that matters

It reduces repeated token burn and stops long conversations becoming the only source of truth.

What to store externally

project state, task lists, accepted decisions, repository notes, operating rules, and workflow-specific memory.

What not to do

keep re-pasting giant blobs of history into chat or expect the model to “just remember” everything indefinitely.

Practical rule: pass summaries, task lists, issue references, and documents between agents — not full conversational histories unless there is a very specific reason.
Operating gospel

These are the principles the proposal should revolve around

This should not just be implied throughout the document. These principles should be concreted in as the core doctrine the implementation will be built around.

Workflow-first, not chat-first. The point is not to give everyone another AI chat surface. The point is to improve real workflows.
Specialised agents over one overloaded agent. Split coordination, execution, review, and support where it creates clarity and control.
Dedicated identities per agent. Keep roles, permissions, and auditability clean.
Human approvals at key risk points. External sends, sensitive actions, merges, and permission expansion should not happen casually.
Evidence beats narration. Agents should report commands, files, tests, artifacts, blockers, and concrete results.
Mission control is part of the product. Visibility, intervention, session health, approvals, failures, and costs need a control layer.
Cost is an operating discipline. Use routing, cheap-first defaults, session narrowing, and central spend tracking.
Memory belongs in systems, not just context windows. Externalize durable state into docs, projects, and structured files.
Autonomy should expand only where trust has been earned. Start bounded, prove leverage, then widen scope deliberately.
Targeted leverage beats universal rollout. This model is strongest in coordination-heavy workflows with clear owners and repeatable patterns.
