From a personal operating system to an organizational operating model

This document should be read as a two-part story. Part 1 shows how the current system works in practice today using Peppermint Butler and BMO. Part 2 shifts from personal productivity into organizational design and explains what would matter if a model like this were implemented more formally inside Memrise.

Reading note: Part 1 is about showing the machine in motion: real workflows, real agents, real interfaces, real examples. Part 2 is about what would need to be true to make the same pattern viable, safe, and useful in a company context.
Part 1 — Current operating system

How the current system works in practice

This first part is deliberately concrete. The point is not to talk about agent architecture in the abstract. The point is to show how a small multi-agent operating system already works today in practice: what it is, what problem it solves, why specialization matters, and how the different parts fit together.

1. Overview

What this system is and why it exists

This system is structured around specialised agents, separate identities, lightweight interfaces, and explicit control layers because the goal is not just to make AI available. The goal is to make it usable, governable, and effective in real workflows.

A single general-purpose agent quickly becomes overloaded. It has to hold too much context, switch between too many responsibilities, and operate across too many systems at once. That increases cost, reduces clarity, and makes failures harder to understand. By splitting work across specialised agents, each agent can operate with a narrower role, a smaller context window, and a clearer operating boundary.

The result is a system that is easier to supervise, easier to scale, and less likely to become an expensive black box.

2. The current setup

Peppermint Butler, BMO, and why the split matters

The current setup uses two primary agents with different roles. Pep handles orchestration, synthesis, routing, and day-to-day operational support. BMO handles technical execution. Keeping those roles separate is what makes the whole thing practical.

Peppermint Butler

Default tier: Tier 1 / Tier 2 · Escalate: Tier 3

  • meeting-note summaries
  • AI news scans
  • inbox triage
  • task routing
  • project setup drafts
  • workflow and account coordination
  • proactive updates and requests

BMO

Default tier: Tier 3 · Escalate: Tier 4

  • coding tasks and implementation
  • refactors and debugging
  • PR support and code review
  • technical documentation
  • repo-linked workflow execution
  • escalation on blockers or risky actions
Why individual accounts?

Dedicated identities make agent-based workflows easier to manage, safer to operate, and clearer to supervise.

Clear separation of roles

Each agent has a distinct function and operating surface, rather than one shared identity doing everything.

Better auditability

Actions can be traced back to the specific agent account that performed them.

Safer permissions design

Access can be scoped by workflow and role, instead of giving one general-purpose identity broad access to everything.

Supports controlled autonomy

Agents can operate independently within defined boundaries, while still escalating sensitive or higher-risk actions.

Cleaner handoffs between agents

A planning agent and an execution agent can work together without collapsing into one overloaded session or identity.

Less operational ambiguity

It is easier to understand who did what, where work lives, and which agent is responsible for the next step.

More resilient workflow design

If one agent drifts, fails, or needs to be reset, it can be managed without disrupting the whole system.

Bottom line: individual accounts turn agents from a shared black box into distinct, traceable operators.
How they work together: Pep shapes work, routes it, and keeps the workflow legible. BMO picks up the execution-heavy technical path. That split is what lets planning and implementation remain connected without turning into one muddled session.
3. Development workflow

From scoping to tracked execution

This is the clearest place to show the system working in miniature. A request comes in, Pep turns it into a structured project and task list, BMO takes the technical path, and the work is tracked through to completion rather than disappearing into an opaque chat log.

Scoping. Pep turns a raw request into something bounded and actionable.
GitHub project creation. The request becomes a visible project workflow rather than an informal memory burden.
Task generation. Work is broken into clear technical steps.
Handoff to BMO. Execution passes to the specialist technical agent.
Tracked execution to completion. Progress is visible through tasks, issues, PRs, and artifacts.
4. Interfaces

Why Telegram and Slack work well in practice

Telegram

Telegram works because it is fast, lightweight, mobile-first, and easy to use as a direct line into the system.

  • quick interaction on the go
  • simple direct requests and updates
  • easy session reset and recovery
  • contained context per agent or thread

Slack

Slack works because it fits into existing operational workflows and supports shared visibility, approvals, and coordination.

  • team visibility
  • human-in-the-loop approvals
  • shared workflow updates
  • coordination alongside GitHub and other systems
5. Control layer / mission control

How the agents are supervised

Once work is distributed across multiple agents, visibility becomes a core operating need. Operators need to be able to see what is running, what has completed, what is blocked, what has failed, and where intervention is required. Without that, the system becomes a black box.

What the dashboard solves

  • live session visibility
  • running workflows and stuck states
  • approvals and exceptions
  • cost and usage alongside activity
  • easier intervention through reset, reroute, or stop
6. What is working / current limitations

What already works and what still feels messy

What is already working well

  • agent specialization keeps roles clearer
  • the development path from scoping to tracked execution is real
  • Telegram and Slack both work as practical interfaces
  • mission control improves visibility and operator trust

Current limitations

  • memory is still weaker than it should be
  • security and permissions are not yet as tight as a company rollout would require
  • visibility is better, but not complete
  • some workflows still feel hand-built rather than formalized
Bridge: The current setup is useful because it shows the model working in practice. But moving from a personal operating system to a team or company context changes the requirements. At that point, visibility, permissions, cost control, governance, and workflow design become much more important.

8. Candidate use cases / agents at Memrise

Translating the pattern, not copying the personalities

Memrise probably should not have Peppermint Butler or BMO by name. But it can adopt the concepts behind them: coordination, execution, and visibility as distinct roles with different model tiers and access boundaries.

Workflow Coordination Agent

Default: Tier 1 / Tier 2 · Escalate: Tier 3

  • support triage and routing
  • planning and task shaping
  • meeting-note synthesis
  • project setup
  • handoffs between people, agents, and systems

Technical Delivery Agent

Default: Tier 3 · Escalate: Tier 4

  • product and coding execution
  • refactors, debugging, PR support
  • technical documentation
  • build/review loops
  • repo-linked implementation work
IR

Insights & Reporting Agent

Default: Tier 1 / Tier 2 · Escalate: Tier 3

  • planning and status visibility
  • executive briefings and summaries
  • operational digests
  • exception spotting and follow-up
  • lightweight review prep

Other candidate workflows

support, marketing / Braze, planning and status visibility, executive briefing / summaries, security / review assistance.

What matters most

The names are less important than the boundaries: who coordinates, who executes, who reports, and what each role is allowed to do.

9. Security and governance

What would need to be true inside Memrise

Controls that matter

  • dedicated identities
  • least privilege
  • approval gates
  • auditability
  • clear draft vs execute boundary
  • hosting model chosen intentionally

Design implications

  • not all actions should execute automatically
  • humans stay accountable for high-impact decisions
  • memory and context should be externalized, not hidden in giant chats
  • autonomy should expand only where trust has been earned
10. Cost control and model routing

Cheap-first routing, clear tiers, and explicit escalation

Cost control should be treated as an operating discipline, not just a model-choice preference. The goal is to reserve stronger models for genuinely difficult work while cheaper models handle routine tasks.

Tier 1 — cheapest / fastest

  • classification
  • tagging
  • daily digests
  • reminders
  • simple summaries
  • lightweight routing
  • inbox triage

Examples: Gemini 2.5 Flash-Lite · GPT-5 mini · Claude Haiku 4.5

Tier 2 — low-cost general work

  • project shaping
  • rough drafting
  • meeting-note synthesis
  • task breakdown
  • lightweight web / docs work
  • agent-to-agent summaries

Examples: Gemini 2.5 Flash · GPT-5 mini · Claude Haiku 4.5 / low-cost Sonnet

Tier 3 — strong default production model

  • most coding work
  • non-trivial planning
  • implementation steps
  • code review
  • debugging
  • quality-sensitive drafting

Examples: GPT-5.4 · Claude Sonnet 4.6 · Gemini 2.5 Pro

Tier 4 — premium / sparse use

  • hardest debugging
  • architecture decisions
  • difficult synthesis
  • large codebase reasoning
  • critical final passes
  • stubborn edge cases

Examples: Claude Opus 4.6 · GPT-5.4 high-reasoning · Gemini 2.5 Pro

Operational pattern: use cheap-first routing, per-agent budgets, spend tracking, narrow contexts, structured handoffs, and resets after milestones rather than letting sessions turn into whales.
11. Who this is for

Best for selected workflows, not for everyone by default

Best for

  • operators and workflow owners
  • product and project leads
  • support and lifecycle-heavy roles
  • teams with repeatable, high-friction processes

Not to be trusted by default

  • for everyone in the company
  • for high-risk actions without review
  • for workflows with unclear ownership
  • for contexts where deterministic behavior is essential
12. Key principles / guiding stars

The principles the adoption model should revolve around

Visibility over black boxes. If the work is distributed across agents, operators need to be able to see what is happening.
Specialization over one giant agent. Split roles where it creates clarity, control, and better handoffs.
Autonomy within boundaries. Agents can move work forward independently, but not without guardrails.
Human review where risk matters. Sensitive actions, merges, external sends, and permission changes should stay gated.
Cost-aware by design. Routing, narrow contexts, and spend tracking should be built in from the start.
Fit into existing workflows. Slack, Telegram, GitHub, and adjacent systems should be part of the operating model, not afterthoughts.
Start narrow, scale carefully. Prove leverage in selected workflows before broadening autonomy.
13. Closing takeaway

This is not a chatbot rollout

This is an operating model for selected workflows. The point is to increase leverage without losing control: show the system working in miniature, then design the cleaner, safer, more governable version that a company context would require.
